# Design: Rename `poc-controller` → `opm-operator`

## Research & Decisions

### Atomic PR vs. phased rollout

**Context**: The rename touches six distinct surfaces (Go module path, CUE catalog module, in-cluster identity, build/CI glue, docs, workspace coupling). A phased approach was considered to shrink each PR.

**Explored**: A phased order that would keep the tree buildable at every step was sketched: (1) Go module path, (2) kustomize namespace + labels, (3) CUE catalog module, (4) task glue and docs. The problem is that step 1 alone requires updating the `PROJECT` file's `repo` field and every Go import — the tree cannot be "partially renamed" without the `go build` graph changing. Splitting the in-cluster identity change from the module-path change would ship a release whose manifests disagree with its module path, which is worse than either endpoint.

**Decision**: Single atomic PR for every in-repo change. Workspace-level changes (root `CLAUDE.md`, `.code-workspace`, `releases/Taskfile.yml`) ship in the same PR because the disk directory rename happens concurrently.

**Rationale**: The rename's value comes from every surface agreeing; intermediate states have no operator audience. Principle VIII (small batch sizes) applies within a focused change — a rename is one concern even though it edits many files. Mechanical `s/poc-controller/opm-operator/g` on import paths plus targeted label/namespace edits is reviewable as a single diff.

### Upgrade strategy for running clusters

**Context**: A Deployment's `spec.selector.matchLabels` is immutable. The current manager Deployment's selector is `control-plane: controller-manager, app.kubernetes.io/name: poc-controller`. Changing the label to `opm-operator` cannot be done with `kubectl apply` against the existing Deployment.

**Explored**: Three options: (a) keep the old `app.kubernetes.io/name: poc-controller` label and rename everything else; (b) introduce a `matchLabels` shim that selects both old and new values during a transition release; (c) delete-then-install the controller.

**Decision**: Option (c). The documented upgrade procedure is:

```
kubectl delete -n poc-controller-system deployment/poc-controller-controller-manager
kubectl delete -n poc-controller-system serviceaccount/poc-controller-controller-manager
kubectl delete clusterrole poc-controller-manager-role poc-controller-metrics-reader
kubectl delete clusterrolebinding poc-controller-manager-rolebinding poc-controller-metrics-binding
kubectl delete namespace poc-controller-system
kubectl apply -f https://github.com/open-platform-model/opm-operator/releases/download/v0.5.0/install.yaml
```

**Rationale**: (a) defeats the purpose of the rename — we would be left with `opm-operator-system` namespace hosting a deployment selecting on a stale label. (b) adds a temporary selector the controller code must honor for one release and then remove — carries rollback complexity for a one-time cutover. (c) is the simplest operation and is safe because: the leader-election `Lease` is transient and gets re-elected on the new Deployment; CRDs are cluster-scoped and not touched by the namespace deletion; user `ModuleRelease` and `BundleRelease` resources live in user namespaces and already carry `app.kubernetes.io/managed-by: opm-controller` (not `poc-controller`), so the renamed controller claims them on first reconcile without edits. The ownership labels were decoupled from the manager identity in ADR-010.

### `PROJECT` file

**Context**: The kubebuilder `PROJECT` file is a generated scaffold with `# Code generated by tool. DO NOT EDIT.` at the top. Editing it directly is unconventional.

**Explored**: Regenerating via `kubebuilder init` / `kubebuilder edit` was considered, but those commands rewrite more than the `projectName` and `repo` fields and would churn the resource list, markers, and layout version.

**Decision**: Hand-edit `projectName: poc-controller` → `opm-operator` and `repo: github.com/open-platform-model/poc-controller` → `github.com/open-platform-model/opm-operator`. Leave every other field untouched. Update the three `path:` entries under `resources:` that carry the module path.

**Rationale**: The "DO NOT EDIT" header is advisory against churn, not a correctness gate. Kubebuilder re-reads `PROJECT` only when scaffolding new resources; for a project with no planned new scaffolds this is safe. Hand-editing keeps the diff surgical.

### Release-please version continuity

**Context**: `release-please-config.json` has `package-name: "poc-controller"` and `.release-please-manifest.json` records `{".": "0.4.4"}`. Changing the package name without carrying the manifest forward would reset release-please's version-tracking heuristics.

**Explored**: Leave `package-name` unchanged (release-please would still produce `v0.5.0` tags on the new repo name), or change both and preserve the manifest.

**Decision**: Update `package-name` to `opm-operator`. Leave `.release-please-manifest.json` at `{".": "0.4.4"}`. The next release-please run emits `v0.5.0` on the renamed repo.

**Rationale**: `package-name` appears in release-please PR titles and changelog section headers; not updating it would keep "poc-controller" visible in every future release PR indefinitely. The manifest tracks version state independently of the package name, so `v0.4.4 → v0.5.0` transitions cleanly.

### Disk directory rename and workspace coupling

**Context**: The workspace root is `/var/home/emil/Dev/open-platform-model`. The repo lives at `poc-controller/`. The root `CLAUDE.md` table, the `open-platform-model.code-workspace` file, the root `Taskfile.yml`, and `releases/Taskfile.yml` all embed `poc-controller` either as a directory path or as an in-cluster identifier for kubectl commands.

**Explored**: Defer the directory rename to a separate follow-up PR, or pair it with the in-repo rename.

**Decision**: Pair. The rename PR is authored after the directory is renamed on disk. The PR includes edits to root-level files that reference the old directory or the old in-cluster identifiers. The GitHub repo rename is performed manually by the user between merging the rename PR and merging the release-please PR it triggers.

**Rationale**: Leaving `poc-controller/` on disk while the repo inside it calls itself `opm-operator` is exactly the kind of divergence this change exists to eliminate. Since the root workspace file references are the only cross-repo coupling, bundling them keeps the rename atomic from the workspace's perspective.

### Go module path bridge

**Context**: A final `v0.4.5` release under `github.com/open-platform-model/poc-controller` with a `retract` directive and a README pointer to the new path was considered as a courtesy to any external Go consumer.

**Decision**: Skip. No external Go consumers are known; the workspace has none. GitHub preserves git-level redirects for clone/fetch. The Go proxy serves existing `v0.4.x` tags under the old path in perpetuity.

**Rationale**: Adding a bridge release for a consumer base of zero is pure overhead.

## Upgrade Procedure

Documented in `README.md` and referenced from the release notes:

1. Export any in-flight `ModuleRelease` / `BundleRelease` CR state (`kubectl get modulereleases.releases.opmodel.dev -A -o yaml > backup.yaml` — optional; CRs are not touched, this is defensive).
2. Delete the old manager:
   ```
   kubectl delete -n poc-controller-system deployment poc-controller-controller-manager
   kubectl delete clusterrole,clusterrolebinding -l app.kubernetes.io/name=poc-controller
   kubectl delete namespace poc-controller-system
   ```
3. Install the new manager:
   ```
   kubectl apply -f https://github.com/open-platform-model/opm-operator/releases/download/v0.5.0/install.yaml
   ```
4. Verify:
   ```
   kubectl -n opm-operator-system rollout status deployment/opm-operator-controller-manager --timeout=120s
   kubectl get modulereleases.releases.opmodel.dev -A
   ```
   Existing `ModuleRelease` resources become `Ready` again on the next reconcile. Their rendered workloads stay up throughout the cutover (the old controller's last successful apply is still live; the new controller re-computes inventory and re-applies).

## Rollback

The reverse of the upgrade procedure: delete the `opm-operator-system` namespace and associated cluster-scoped RBAC, then reinstall a `v0.4.x` controller from its `install.yaml`. User CRs remain valid because the CRD schema is unchanged. Rollback is cheap because the controller is stateless beyond `status` (which is reconstructed from live cluster state).

## Reconcile Phase Impact

None. The reconcile loop (Source → Render → Apply → Prune → Status) is unaffected. The Field Manager for SSA remains `opm-controller`. The managed-by label value on rendered user resources remains `opm-controller`. Catalog render injects `#context.#runtimeName` with the same value as before. No CRD schema field is added, removed, or renamed. DeepCopy regeneration is a no-op because `api/v1alpha1` contains no string literal referencing `poc-controller`.

## Validation Plan

Ordered gates, all of which must pass locally before the PR is opened and re-pass in CI:

1. `task dev:manifests dev:generate` — regenerate CRDs and DeepCopy; diff should be empty.
2. `task dev:fmt dev:vet` — formatting and vet pass.
3. `task dev:lint` — golangci-lint clean.
4. `task dev:test` — envtest unit and integration suites green. The `test/integration/reconcile/runtime_identity_test.go` suite exercises the managed-by contract; passing it confirms the runtime identity decoupling from the manager rename.
5. `task dev:e2e` — Kind e2e with the renamed namespace, SA, and RBAC. Requires updating the hardcoded constants in `test/e2e/e2e_test.go` in the same commit.
6. `task operator:installer` — verify `dist/install.yaml` renders with the new namespace and name prefix.
7. Smoke test: `task operator:controller:install IMG=controller:latest` against a throwaway Kind cluster, apply a sample `ModuleRelease`, confirm the rendered `ConfigMap` carries `app.kubernetes.io/managed-by: opm-controller` (unchanged) and that inventory reports it under the new namespace.
